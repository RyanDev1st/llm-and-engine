# v1.2 SFT dataset — alignment audit

## Status

**NOT aligned. Do not train as-is.** The corpus trains a Claude-Code-style *skill/plugin harness* (`load_skill` + `skills_index` + plugin gating + multi-step tool chains) that does **not** match the deployed chess backend, and the harness context it is built around is **discarded by the training loader**. On top of that the chess content is fabricated and the eval split leaks. Numbers below are from `data/sft/v1_2_train.jsonl` (44,983 rows) + `v1_2_val.jsonl` (5,019), cross-referenced to the real backend.

## Scope

Goal being audited against: the agent is a **tool-router/narrator** for chess — it routes user intent to the backend's tools and narrates results; it does not compute chess. So "aligned" = the trained behavior must (a) call the tools the backend actually exposes, (b) in the format the injected system prompt defines, (c) with truthful tool results, (d) and a holdout that measures generalization.

## Evidence

### A. Contract mismatch — dataset trains a tool surface the environment doesn't expose

1. **`load_skill` is not a backend tool.** `ToolExecutor._dispatch` (`src/llm/backend/tools.py:52-77`) implements exactly 10: `board_state, move, undo, legal_moves, list_pieces, ask_chessbot, eval, best_move, review_move, threats`. Unknown name → `error: invalid_syntax` (`tools.py:44,77`). The dataset opens **100% of rows (44,983/44,983)** with `<tool>load_skill ...>`. At serving, the agent's trained first action fails.
2. **43% of all tool calls go to tools the backend can't run:** 47,035 / 110,039. Breakdown: `load_skill` ×45,667, `normalize_human_chat` ×684, synthetic `tool_*` (universality fixtures).
3. **System prompt ↔ data disagree on the tool list.** Both training loaders inject `SYSTEM_PROMPT` (`data_pipeline.py:14-28`, `jsonl_loader.py:9-23`) which lists **9 tools — no `load_skill`, no `board_state`** (`system_prompt.py:14-23`). The data calls both heavily. The model is trained to call tools its own system prompt omits.
4. **Mode-2 contradiction.** `SYSTEM_PROMPT` says: *"RESPONDING TO A TOOL RESULT … you must NOT emit another tool call. Just talk."* The data has **59% (65,056/110,039)** assistant-after-tool turns that emit another tool (the `load_skill→board_state→move` chains). The corpus systematically trains the exact behavior the injected prompt forbids.
5. **The skill-routing signal is thrown away.** `skills_index` is present in **44,983/44,983** envelopes, plus `tool_manifest` and `plugin_context` — but the loaders serialize **only `messages`** + a fixed system prompt. The model never sees the index it "selected from," nor the manifest/plugin/enabled state. So skill-selection, plugin gating, dynamic-tool-schema, and adversarial-injection context are **all invisible during training**. The model can only rote-memorize "emit `load_skill name=chess-coach`."

### B. Content correctness — of what little IS in the training text

6. **59% of move rows are illegal** (5,542/9,371) for their `position_fen`, with a fabricated `success: e4` tool turn. **Move monoculture:** 1 distinct move (`e4`) across all 9,371.
7. **board_state fidelity drift:** 26,507/27,875 board_state results embed `fen=` in "basic" (real "basic" = `turn,last_move,check,legal_count`, no fen — `tools.py:82-83`); **12,621 report the wrong side** (`turn=` ≠ FEN side-to-move).
8. **93% val leakage:** 4,679/5,019 val final-targets appear verbatim in train → the eval metric is meaningless.
9. **65% persona-opener finals** (29,161/44,983): "No fluff", "Cutting to it", "Plain take", etc. — trains tone, which is explicitly out of scope for this SFT.

### C. What is sound (keep)

- One tool call per assistant turn: **0** turns with >1 call (matches the prompt's max=1).
- Reject pool: 11 balanced anti-patterns (~682 each).
- Tool-call *sequence* structure is rich (653 distinct shapes) — the scaffolding shape is right; it just points at the wrong tool contract.

## Root causes

- Generator `src/llm/llm_dataset/v1/renderer/chess.py` hardcodes `e4`/`success: e4`/`turn=white`.
- The harness layer (`load_skill`, `skills_index`, `tool_manifest`, `plugin_context`) was designed but (a) the backend never grew a `load_skill`/skills-manifest tool surface, and (b) the loaders (`data_pipeline.py`, `jsonl_loader.py`) never serialize the envelope into the model's context.
- `validate.py` `engine_grounded` rule never checks move legality.

## The decision this forces (A vs B) — must choose before any Phase-1 data fix

**Option A — align DATA to the current backend (ship-now, smaller).**
- Remove `load_skill` + the skills/plugin envelope from training rows; train pure routing/narration over the real 10 tools using the real one-tool-then-narrate contract.
- Ground moves to legal SANs, fix board_state to `tools.py` format, regenerate, de-leak split, drop persona openers.
- Pro: matches what the chess web app can actually run today. Con: drops the "generalize to skills/plugins" ambition.

**Option B — align BACKEND + loader to the data (bigger, matches the universality goal).**
- Backend grows a real harness: a `load_skill` tool + a skills/plugin manifest; server injects `skills_index`/`tool_manifest` into the system prompt; allow multi-step Mode-2 chaining (rewrite `SYSTEM_PROMPT`).
- Loader must serialize `skills_index`/`tool_manifest`/`plugin_context` into the training context (else the signal stays untrained).
- Still must fix legality/board_state/leak/personas.
- Pro: realizes the skill-routing agent in memory `chess-agent-sft-generalization`. Con: substantial backend + loader work.

## Next

1. **User/Codex decision: Option A or B.** Everything downstream depends on it; `implementation.md` Phase 1 currently fixes only move-grounding and must be amended to carry this decision.
2. If **A**: edit `renderer/chess.py` to drop `load_skill`/envelope + ground moves; align `SYSTEM_PROMPT` and the 10-tool surface; regenerate; re-audit (target: 0 non-backend tool calls, 0 illegal moves, <1% val leak, 0 persona openers).
3. If **B**: implement backend `load_skill` + manifest injection; serialize the envelope in the loader (`data_pipeline.py`); rewrite `SYSTEM_PROMPT` for skill-first multi-step; then ground content + de-leak; re-audit.
4. Either path: add the `validate.py` legality gate + a loader test asserting the training text contains every tool the rows call (prevents silent contract drift recurring).
